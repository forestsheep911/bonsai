# 盆景 (bonsai) · 项目愿景

> 一个个人开发者的"项目盆景园"——把脑子里、硬盘里、各个 Git 仓库里那些
> 半成品、原型、MVP、在跑的小工具，像养盆景一样精心摆出来：
> 不必每盆都长成参天大树，每一盆都有它独特的姿态，值得被欣赏与记录。
>
> _项目代号：`bonsai`。盆景隐喻贯穿整个产品语言（状态机 / 文案 / 视觉）。_

---

## 1. 一句话愿景

**为 AI 时代的独立开发者，提供一个轻量、可对外的"在做项目"看板，
让每个想法从灵光一闪到上线运行的全过程都能被记录、被追踪、被展示。**

---

## 2. 为什么要做这件事（Why）

### 2.1 时代背景
进入 AI 辅助编程时代后，独立开发者的生产模式发生了本质变化：

- **想法→原型的距离极短**：一个周末就能跑出 MVP，一晚上就能搭起 Demo。
- **项目数量爆炸**：从过去"一年一个像样的项目"，变成"一个月好几个想法落地"。
- **生命周期参差不齐**：有的项目两天后被放弃，有的悄悄运行了一年还在赚钱，
  有的卡在 70% 死活不上线。

### 2.2 现实痛点
开发者自己面临的问题：

| 痛点 | 具体表现 |
| --- | --- |
| **看不清自己的全景** | 项目散落在 GitHub、本地、Vercel、各种笔记里，没法一眼看到全貌 |
| **想法容易被遗忘** | "上次那个挺有意思的小工具叫什么来着？" |
| **状态难以维护** | 哪个还在更新？哪个已经死了？哪个其实在线上跑着但忘了？ |
| **难以对外介绍** | 简历/个人主页太正式，GitHub 太技术，没有一个适合"展示在做的事"的地方 |
| **缺乏复盘视角** | 半年后回头看不到自己的轨迹和成长 |

观众/潜在合作者/雇主面临的问题：

- GitHub 只能看到代码，看不到"项目的故事"和"现在的状态"。
- 个人博客重内容轻索引，不适合"快速浏览这个人最近在折腾什么"。
- 没有一个地方让我快速回答："这个开发者最近在做什么？做到哪一步了？"

### 2.3 我们要填补的空白
**GitHub 偏重代码，Product Hunt 偏重发布，个人博客偏重叙述——
中间缺一个"持续展示在做的事"的轻量化货架。**

---

## 3. 目标用户

### 3.1 第一用户（自己）
- AI 时代的独立开发者 / Indie Hacker / 副业玩家
- 同时维护 3 个以上"在做"或"在跑"的项目
- 愿意公开展示进展，但不想为每个小项目都做一个独立 Landing Page

### 3.2 第二用户（观众）
- **同行开发者**：看到有意思的小项目，可能复用、参考、合作。
- **潜在雇主/客户**：希望快速了解"这个人最近的真实产出"。
- **未来的自己**：半年后回顾自己的折腾轨迹。

## 4. 核心场景

### 场景 A：一个新想法诞生
> 周五晚上，我突然想做一个 X，先创建一个项目卡片，状态 = `idea`，
> 写两句话先存着，省得忘了。

### 场景 B：周末出了个原型
> 把状态从 `idea` 改成 `prototype`，贴上 demo 截图、在线链接和仓库链接。

### 场景 C：朋友问我"你最近在搞什么"
> 直接甩一个网址过去：`bonsai.<我>.com`，
> 比口头描述清楚 10 倍，比简历轻松 100 倍。

### 场景 D：项目悄悄上线运行了
> 状态变成 `live`，加上"用户数 / MRR / 访问量"等关键指标，
> 让别人（也让自己）看到它真的在为世界产出价值。

### 场景 E：项目失败/搁置
> 不删除，标为 `paused` 或 `archived`，写一段"为什么停了"，
> 这是最有价值的复盘资产。

### 场景 F：年终回顾
> 一年下来，看一眼时间线：12 个 idea，6 个原型，3 个上线，1 个死了——
> 这就是真实的我。

## 5. 关键概念（Domain Model 雏形）

### 5.1 项目（Project）
最小展示单位，包含：

- **基础信息**：名称、一句话简介、封面图、标签
- **状态**：见下方"项目状态机"
- **链接**：仓库、Demo、官网、文档、相关博客
- **里程碑**：关键事件流（创建、开源、上线、改版、归档…）
- **指标**（可选）：用户数、收入、Star、访问量等
- **故事**：长文 markdown，记录想法来源、技术选型、踩坑、思考

### 5.2 项目状态机（建议初版）

```
idea ──► prototype ──► mvp ──► live ──► mature
  │          │          │        │
  └──────────┴──────────┴────────┴──► paused / archived
```

| 内部状态 | 盆景版显示词 | 含义 |
| --- | --- | --- |
| `idea` | 🌱 **种子** | 只是个想法，可能只有一段描述 |
| `prototype` | 🌿 **嫩芽** | 跑通了核心 demo，丑陋但能用 |
| `mvp` | 🪴 **幼苗** | 可发布的最小可用版本 |
| `live` | 🌳 **成型** | 正在线上运行，有真实用户 |
| `mature` | 🎋 **老桩** | 稳定运营，迭代变缓 |
| `paused` | 💤 **休眠** | 暂停浇水，未来可能继续 |
| `archived` | 📦 **入库** | 不再打理，但留作纪念 |

> 注意：`内部状态` 是数据库与 API 字段（英文小写），不会变；
> `盆景版显示词` 仅是 UI 展示层的命名，可独立调整。
> 最终 emoji 是否上线由视觉设计阶段决定。

### 5.3 时间线（Timeline）
所有项目状态变更、里程碑事件汇总，
形成"个人开发者的盆景培育日志"。

## 6. 设计原则

1. **轻量优先**：写一个项目卡片不应该比写一条朋友圈更累。
2. **公开为默认**：所有项目默认对外可见，私密是例外。
3. **状态比文档重要**：用户最关心的不是"这是什么"，而是"它现在怎么样"。
4. **演进可见**：项目的轨迹（时间线）和当前状态同等重要。
5. **可索引可分享**：每个项目有独立 URL，方便贴到任何地方。
6. **诚实展示**：失败的项目和成功的项目同等值得被展示。
7. **数据可携带**：项目数据应能以 markdown / json 形式导入导出，避免锁死。

## 7. 与现有方案的差异化

| 方案 | 它擅长什么 | 它不擅长什么 |
| --- | --- | --- |
| **GitHub Profile / README** | 展示代码、贡献图 | 不展示"非代码"项目状态、运营指标 |
| **个人博客** | 长文叙述、SEO | 不适合"快速看全景"和"持续状态更新" |
| **Notion 公开页** | 灵活 | 太通用、缺乏"项目"语义、加载慢 |
| **Product Hunt** | 一次性发布 | 不适合"在做中"和"已死"的项目 |
| **Linktree 类** | 链接聚合 | 没有项目维度的状态和故事 |
| **盆景 (bonsai)** | **持续、轻量、有状态地展示在做的项目** | 不做协作、不做工单、不做代码托管 |

## 8. 范围边界（v1 做 / 不做）

### v1 一定做
- 项目的增删改查（CRUD）
- 项目状态、标签、链接、封面、长文故事
- 项目列表页（按状态/标签筛选、按更新时间排序）
- 项目详情页（独立 URL，适合分享）
- 全站时间线视图
- 响应式（手机能看）

### v1 不做（明确放到 v2+）
- 多用户/SaaS 化（v1 就是单人站）
- 评论、点赞、订阅
- 协作、邀请成员
- 数据看板、复杂统计
- 自动从 GitHub 同步
- RSS / Atom 输出
- 周报 / 月报自动汇总
- 国际化（先做中文，必要时再加英文）

### 技术栈（2026-05-01 晚 · 全栈 Azure 化敲定）
- **前端**：Vite + React 19 + TypeScript ✅
- **UI 库**：shadcn/ui + Tailwind CSS（Radix UI 为底） ✅
- **后端 API**：Azure Functions（Node 22, TypeScript, v4 Programming Model）✅
- **数据库**：Azure Cosmos DB（NoSQL 文档库，Free Tier 优先）✅
- **数据校验**：Zod（替代传统 ORM）✅
- **图片存储**：Azure Blob Storage（复用 SWA 自带 Storage Account）✅
- **部署平台**：Azure Static Web Apps（Free Tier）✅
- **鉴权**：SWA 内置 GitHub OAuth + 自定义角色白名单 ✅
- **CI/CD**：GitHub Actions（`Azure/static-web-apps-deploy@v1`）✅
- **包管理**：pnpm ✅

> 这套技术栈是 2026-05-01 多轮迭代后的最终结论。
> 关键转变：从早晨初定的 "Next.js + Vercel + Postgres" 改为 "Vite + Azure SWA + Cosmos"，
> 核心驱动力是用户对"Vercel 触红线被墙"的顾虑，以及已存在的 factory-soon 项目验证过同款技术栈。
> 详细推理与各项变更原因见 [`DECISIONS.md`](./DECISIONS.md) 各 T-xxx 条目。

## 9. 成功的样子（衡量标准）

**对自己：**
- 每个项目变更，1 分钟内能更新到站上。
- 任何时候打开主页，都能一眼回答"我现在手头有什么"。
- 一年后回看，能清楚地讲出每个项目的故事。

**对别人：**
- 别人花 30 秒就能了解"这个人最近在做什么、做到哪了"。
- 至少有一个项目因为被这个站看到，带来了实际的连接（合作 / 用户 / Offer）。
