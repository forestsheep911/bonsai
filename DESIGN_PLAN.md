# Bonsai 设计落地计划

## Summary

本文档记录从当前简陋原型走向设计稿方向的执行计划。

计划覆盖公开首页、项目详情页、全站时间线、后台编辑器和移动端公开视图，目标是把 `design/generated-images/` 中的 5 张概念图转化为可实现的 v1 UI 路线图。

本文档定位为 **设计落地全局计划**：

- 不替代 `VISION.md`
- 不替代 `DECISIONS.md`
- 不展开 Cosmos/API 深度实现
- 当前阶段允许继续使用 mock 数据

## 当前差距评估

当前实现已经具备基本页面和数据模型，但整体仍接近“shadcn 默认卡片 + mock 数据”的原型状态。

大致完成度判断：

- 数据结构 / 内容能力：约 60%
- 公开首页视觉：约 30%
- 项目详情页视觉：约 45%
- 时间线页：0%
- 后台编辑器：0%
- 移动端体验：约 25%

主要差距不在颜色，而在产品骨架：

- 首页缺少状态统计、筛选、搜索、排序和最近更新侧栏。
- 详情页缺少更完整的产品壳、侧栏信息层级和更强的项目展示感。
- 时间线页尚未实现。
- 后台编辑器尚未实现。
- 移动端目前只是响应式排布，尚未形成专门的信息层级和交互。

## 设计落地原则

1. **产品即首页**  
   第一屏直接展示项目园本体，不做营销式 landing page。

2. **状态优先**  
   项目状态、最近更新和生命周期位置要比长篇说明更醒目。

3. **轻量但完整**  
   UI 要让“1 分钟更新一个项目”成为可能，避免复杂配置和重后台。

4. **盆景隐喻克制使用**  
   使用苔绿、暖白、浅陶土、细线条和植物意象；避免做成插画花园或装饰过重的页面。

5. **信息密度适中**  
   桌面端要适合快速扫描，移动端要适合直接把链接发给朋友、雇主或潜在合作者。

6. **先骨架，后接真实数据**  
   先把视觉与交互骨架落地，数据可以继续来自 `mockProjects`；API/Cosmos 接入放到后续数据层计划。

## 分阶段实施计划

### Phase 1：公开首页重构

目标：让首页明显接近 `01-public-garden` 设计稿，成为真正的“项目盆景园”。

主要改动：

- 顶部导航升级为产品壳：品牌、搜索入口、新建项目入口、个人入口占位。
- 增加个人介绍区：头像 / 盆景图、身份说明、简短标签。
- 增加状态统计条：全部、种子、嫩芽、幼苗、成型、休眠等状态数量。
- 增加筛选工具栏：搜索、状态筛选、标签筛选、排序、视图切换占位。
- 项目卡片改为更紧凑、更接近设计稿的信息结构。
- 桌面端增加最近更新侧栏，展示最近 milestone 或 updatedAt。

验收标准：

- 访问 `/` 时，30 秒内能看懂“这个人最近在做什么、做到哪了”。
- 首页不再像默认组件堆叠，而是有明确产品壳和项目园结构。
- 项目状态统计能从当前 mock 数据计算得出。
- 移动端不出现横向滚动或文字重叠。

### Phase 2：项目详情页增强

目标：让详情页接近 `02-project-detail` 设计稿，成为可以分享给他人的项目介绍页。

主要改动：

- 统一详情页与首页的导航和视觉系统。
- 标题区突出项目名、状态、summary 和更新时间。
- 封面区更像真实产品预览，不只是普通图片容器。
- 右侧栏强化链接、指标、里程碑时间线。
- Markdown 故事排版更适合阅读，保持开发者文档感。
- 底部元信息保留创建时间、更新时间、标签等轻量信息。

验收标准：

- 项目详情页能清楚回答：这是什么、现在什么状态、有无 demo / repo、最近发生了什么。
- 右侧栏信息密度接近设计稿，但不显得拥挤。
- Markdown 内容在桌面和移动端都可读。

### Phase 3：全站时间线页

目标：新增 `/timeline`，实现 `03-global-timeline` 设计稿中的“培育日志”。

主要改动：

- 从所有项目的 `milestones` 汇总事件。
- 按月份分组展示。
- 支持基础筛选占位：状态、事件类型、标签。
- 顶部展示概览统计：想法、原型 / 幼苗、成型 / 上线、归档项目。
- 每条事件可跳转到对应项目详情页。

验收标准：

- `/timeline` 能从 mock 数据渲染跨项目事件流。
- 同一个项目的状态变化和 milestone 能被清楚识别。
- 页面能体现“个人开发者的盆景培育日志”。

### Phase 4：后台编辑器壳

目标：新增 `/admin/projects/:id`，实现 `04-admin-editor` 设计稿中的低摩擦编辑体验。

主要改动：

- 后台使用独立布局：左侧导航、顶部路径、底部保存条。
- 表单包含名称、summary、状态、标签、链接、封面、项目故事。
- 状态使用 segmented control。
- 标签和链接支持静态 mock 编辑 UI。
- Markdown 编辑器与预览并排展示。
- 保存按钮先做 UI 状态，不要求接入真实持久化。

验收标准：

- 用户能在一个页面中看到项目 CRUD 所需的核心字段。
- 编辑器看起来支持“1 分钟更新一个项目”。
- Markdown 预览与详情页渲染风格一致。

### Phase 5：移动端公开视图优化

目标：接近 `05-mobile-public` 设计稿，不只是桌面布局缩小。

主要改动：

- 移动端顶部导航简化为品牌 + 菜单。
- 状态筛选变为横向 chips。
- 项目列表改为更适合手机的紧凑单列。
- 支持一个展开态项目卡片，展示链接和最近更新。
- 页脚保持轻量。

验收标准：

- 390px 宽度下所有文字不溢出、不重叠。
- 项目列表适合手指点击。
- 一个陌生访客在手机上能快速理解项目状态和入口。

## Public Interfaces / Types

本计划不要求新增或修改运行时 API。

后续实现时默认复用现有领域模型：

- `Project`
- `ProjectStatus`
- `ProjectLink`
- `ProjectMetric`
- `ProjectMilestone`

优先复用 `src/domain/project.ts`，不要为了 UI 单独创建一套不兼容的数据结构。

## Test Plan

每个阶段完成后至少运行：

```bash
pnpm check
pnpm typecheck
pnpm build
```

涉及 API 或后端文件时同时运行：

```bash
pnpm --dir api typecheck
pnpm --dir api build
```

视觉验收：

- 桌面端检查首页、详情页、时间线、后台编辑器。
- 移动端检查首页和项目列表。
- 对照 `design/generated-images/` 中对应概念图，确认信息层级和视觉方向接近。
- 确认无明显文字溢出、重叠、横向滚动。

## Assumptions

- 本计划文件固定为 `DESIGN_PLAN.md`，放在仓库根目录。

- 下一步实现优先级为：公开首页重构 → 项目详情增强 → 时间线页 → 后台编辑器 → 移动端优化。
